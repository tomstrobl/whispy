# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

`whispy` is a config-driven Python toolkit for running listening tests / perceptual experiments with PyQt6 UIs (MUSHRA-like drag-and-drop rating, N-AFC, questionnaires, info screens) and audio playback via `sounddevice`/`pyfar`.

There is no `README.md`; `AGENTS.md` is another AI-facing guide that overlaps with this file â€” keep both in sync if architecture changes.

## Commands

- Install in editable mode: `pip install -e .`
- Each test comes in two notebook forms (all in `examples/`, kept flat so relative paths to `../configs` and `demo_stimuli/` work): a minimal **building block** `building_block_<test>.ipynb` (config explanation + run + save, nothing else â€” the copy-paste source) and a finished **full experiment** `full_experiment_<test>.ipynb` (welcome screen â†’ consent questionnaire â†’ test â†’ thank-you screen). Tests: `staircase_n_afc` (adaptive staircase driving N-AFC), `abx` (ABX discrimination, fixed trial list â†’ percent correct), `drag_and_drop_mushra` (MUSHRA), `scale_testing` (attribute rating: one stimulus per screen, rated on stacked Likert-button/slider scales; building block only, no full-experiment notebook yet). `examples/building_block_consent.ipynb` runs the consent questionnaire that produces the anonymous `participant_id`; `examples/building_block_welcome_and_thanks.ipynb` runs the two framing info screens (`configs/welcome.yml` before everything, `configs/thanks.yml` after). There is no standalone single-trial N-AFC notebook; the staircase notebooks are the N-AFC example (they drive `whispy.ui.NAFC`).
- Demo stimuli live under `examples/demo_stimuli/<test>/` (`mushra/`, `staircase/`, `abx/`, `scale_testing/`); each folder holds that notebook's WAVs plus its generator. Regenerate by running the generator from inside its folder, e.g. `python examples/demo_stimuli/mushra/generate_mushra_stimuli.py` (`staircase/generate_staircase_stimuli.py`, `abx/generate_abx_stimuli.py`, `scale_testing/generate_scale_testing_stimuli.py`).

There is currently no test suite, linter, or formatter configured in `pyproject.toml`.

## Core data flow

1. `whispy.ExperimentScheduler(experiment=...)` reads the `experiment:` block of a MUSHRA config (`experiment=` takes the block directly, or the whole combined config and extracts it) and (via the private `_course()` function in `whispy/experiment_scheduler.py`) randomizes blocks/sections/conditions and yields a list of "screen" dicts: `{block, section, reference, test, block_changed, section_changed, attribute, block_name, section_name, progress}`. `experiment=` is required (no package default).
2. Each screen dict is passed to a UI class as `screen=...`:
   - `whispy.ui.DragAndDropMUSHRA(screen=...)` â€” MUSHRA-like drag-and-drop rating.
   - `whispy.ui.NAFC(screen=...)` â€” N-alternative forced choice (screen also carries `test` as a list of choice IDs, plus `correct`/`trial_id`).
   - `whispy.ui.ABX(screen=...)` â€” classic ABX discrimination (screen carries `a`/`b` stimulus IDs, optional `x`; the UI plays A/B/X and the participant answers whether X matches A or B). Modeled on `NAFC` (config-driven, keyboard, window-size scaling, shared-host reuse across trials).
   - `whispy.ui.ScaleTest(screen=...)` â€” rating scales (screen carries a single `stimulus` ID; one play button plus the config's `questions:` stacked below, each a Likert-style button row or a slider). Modeled on `NAFC` like `ABX`. The scheduler yields section screens with a `test` list, so the notebook expands them into one-stimulus screens (recomputing `progress`).
3. `DragAndDropMUSHRA` looks up rating-scale metadata from the `attributes:` block of its experiment config (keyed by `screen["attribute"]`) and layout/theme from the `ui:` block (`configs/drag_and_drop_mushra.yml`); `NAFC`/`ABX` read their layout/theme from their own config (`configs/n_afc.yml`, `configs/abx.yml`) and take the on-screen task prompt inline from the `screen` dict.
4. Tile/button activation triggers playback through a `StimuliHandler` (default `whispy.interfaces.SoundDevice`), keyed by the stimulus IDs in `screen["reference"]` / `screen["test"]`, which map to entries in the config's `SoundDevice:` block.
5. After the window closes, `get_results()` returns a `pandas.DataFrame`; results from multiple screens are accumulated by passing the running DataFrame back in (`DragAndDropMUSHRA.get_results(results)`).

## Configuration conventions

- `whispy.utils.read_config()` (thin `yaml.safe_load` wrapper) is the single shared config loader used everywhere. It passes an already-loaded `dict`/`list` straight through, so a single combined config can be read once and either handed whole to a consumer that extracts its own blocks (e.g. `DragAndDropMUSHRA(drag_and_drop_mushra=cfg)`, which pulls both `ui:` and `attributes:`) or sliced per consumer (e.g. `ExperimentScheduler(experiment=cfg["experiment"])`).
- `whispy.utils.load_design()` returns the global theme from `configs/design.yml` merged with optional per-UI overrides (`load_design(overrides)`, where `overrides` is a dict or YAML path; `None`-valued keys are ignored). Every UI loads its look this way so all listening tests share one theme by default.
- Every config-consuming class resolves a default path relative to its own file's `FILEPATH` if no explicit path is passed, so configs can be overridden per-call without touching package defaults.
- `configs/design.yml`: the single source of truth for the shared look (window size, colors, fonts, button geometry/colors) AND the shared test-layout keys (`content_area_size`, `content_top_share`, `submit_hint_fontsize`/`submit_hint_spacing`, `submit_button_fontsize`, plus per-UI geometry groups: ABX's `x_button_gap`/label fontsizes, MUSHRA's `tile_fontsize`, all ScaleTest question/scale keys). The experiment configs' `ui:` blocks carry only wording (prompts, hints, button texts, `progress_text`), behavior (`show_progress`, autoplay), and deliberate per-experiment overrides of design keys (e.g. ABX's narrower `content_area_size`). The questionnaires keep their own `question_fontsize`/`question_spacing` (same key names as ScaleTest's design keys, different values â€” do not remove those overrides). It carries both the MUSHRA tile-state keys (`button_color_initial/clicked/active`) and the N-AFC button-state keys (`button_background_color`, `button_selected_*`), driven from one palette so the two UIs match. It also carries the `screen:` key (`"primary"` default, `"cursor"`, an index, or a screen name) selecting which monitor every window of an experiment opens on; `_BaseUIWindow._show_host_window()` pins each window to that screen (resolved failsafe by `whispy.ui.base.resolve_screen()` â€” invalid values fall back to the primary screen), an already-visible host keeps its current screen, and each UI reads a per-config override from its merged design (`self._screen_setting`).
- The `experiment:` block (a list of `block -> [block_name, section -> {section_name, attribute, reference, test}]`, inside each MUSHRA config, e.g. `configs/drag_and_drop_mushra.yml`) is consumed by `ExperimentScheduler`. There is no standalone `experiment.yml`: `ExperimentScheduler(experiment=...)` takes the block directly (`cfg["experiment"]`) or the whole combined config (it extracts the `experiment:` block), and is required (no package default).
- The `attributes:` block (inside each MUSHRA experiment config, e.g. `configs/drag_and_drop_mushra.yml`) maps attribute name -> `{task, description, values, labels, neutral_value}` â€” the rating-scale definition shown by `DragAndDropMUSHRA`, keyed by `screen["attribute"]`. It is the only place rating scales are defined; there is no standalone `attributes.yml` (`DragAndDropMUSHRA` pulls the block from its combined config, or accepts an explicit `attributes=` mapping/path that overrides it). `NAFC`/`ABX` do not use attributes at all â€” they take their on-screen prompt from a `task` set directly on the `screen` dict.
- `configs/staircase_n_afc.yml`: a single self-contained config for the `examples/building_block_staircase_n_afc.ipynb` experiment. One file feeds three consumers: `SoundDevice` (its `SoundDevice:` block), `NAFC` (its `test:`/`ui:` blocks, passed as `n_afc_config`), and the notebook (`trial:` = odd-one-out spec, `staircase:` = adaptive-rule kwargs for `whispy.Staircase`). Only the look (`design.yml`) lives elsewhere.
- `configs/abx.yml`: a single self-contained config for `examples/building_block_abx.ipynb` (ABX). One file feeds `SoundDevice` (its `SoundDevice:` block), `ABX` (its `test:` = `shuffle_ab` / `ui:` blocks, passed as `abx_config`), and the notebook (`trial:` = prompt/metadata, `pairs:` = list of `{a, b, name}` comparisons, `experiment:` = `repetitions`/`shuffle_trials`/`seed`). Demo WAVs are generated by `examples/demo_stimuli/abx/generate_abx_stimuli.py`.
- `configs/scale_testing.yml`: a single self-contained config for `examples/building_block_scale_testing.ipynb` (scale testing). One file feeds `SoundDevice` (its `SoundDevice:` block), `ScaleTest` (its `ui:` and `questions:` blocks, passed as `scale_test_config`), `ExperimentScheduler` (its `experiment:` block â€” sections need only `section_name` + `test`; `reference`/`attribute` are optional scheduler-wide since scale testing omits them), and the notebook (`trial:` = the shared task prompt). Each `questions:` entry is `{id, prompt, scale_type, interaction_method, scale_range, labels}`: `interaction_method` is `buttons` (one selectable button per integer step), `slider`, or `bipolar_slider` (negative-to-positive range, zero tick in the middle, no directional fill; default range [-5, 5]); `labels` with two entries are endpoints, with one-per-step (buttons only) they go under each button, and with three entries (bipolar_slider only) they are `[left, center, right]`; parsing is failsafe (broken entries fall back to a 1â€“5 button scale). Demo WAVs are generated by `examples/demo_stimuli/scale_testing/generate_scale_testing_stimuli.py`.
- The `SoundDevice:` block (handler-scoped â€” top-level keys like `SoundDevice:`, `OSCHandler:`) maps stimulus IDs to playback info (e.g. `file:` for `SoundDevice`) and lives inside each experiment's combined config (`drag_and_drop_mushra.yml`, `staircase_n_afc.yml`, `abx.yml`). Stimulus IDs are saved into experiment results â€” prefer descriptive names. There is no standalone `stimuli.yml`: `SoundDevice(stimuli=..., base_dir=...)` requires both an explicit config (a path or dict containing a `SoundDevice:` block) and the folder holding the WAVs â€” it raises a clear `ValueError` if either is `None`.
- `configs/drag_and_drop_mushra.yml`: a single self-contained config for `examples/building_block_drag_and_drop_mushra.ipynb` (MUSHRA), mirroring `abx.yml`/`staircase_n_afc.yml`. One file feeds three consumers: `SoundDevice` (its `SoundDevice:` block), `DragAndDropMUSHRA` (the whole file, passed as `drag_and_drop_mushra` â€” it extracts both the `ui:` layout block and the `attributes:` rating-scale block), and `ExperimentScheduler` (its `experiment:` block). The notebook reads the file once, passes it whole to `DragAndDropMUSHRA`, and slices off `experiment:` for the scheduler. `DragAndDropMUSHRA` accepts either this combined file (it extracts `ui:` and `attributes:`) or a flat UI config plus an explicit `attributes=`; colors/fonts come from `configs/design.yml`.
- `configs/n_afc.yml`: per-UI layout/behavior (`window_size`, `content_area_size`, autoplay, N-AFC `test:`/wording). The package default for `NAFC` when constructed without an explicit `n_afc_config`; no example notebook exercises it anymore (the staircase N-AFC example passes its own combined `staircase_n_afc.yml` as `n_afc_config`). Colors/fonts are inherited from `configs/design.yml`; any theme key may still be overridden here. `window_size: "fullscreen"` is a special-cased string.
- Font sizing: every test exposes one `*_fontsize` key per on-screen text, fully independent of each other; the defaults live in `configs/design.yml` and any of them can be overridden in a test's `ui:` block (N-AFC/ABX: `task_fontsize`, `submit_hint_fontsize`, `button_fontsize`, `submit_button_fontsize`; ABX additionally `listen_label_fontsize`/`answer_label_fontsize` and a `hint_fontsize` group fallback for the three explanatory texts, which N-AFC's `submit_hint_fontsize` also falls back to; MUSHRA: `task_fontsize`, `fontsize` for the scale labels, `tile_fontsize` for the draggable tiles falling back to `button_fontsize` for the Reference/Stop/Continue buttons; Questionnaire: `section_fontsize`/`question_fontsize`; ScaleTest: `task_fontsize`, `question_fontsize`, `scale_label_fontsize`, `button_fontsize`, `play_button_fontsize`, `submit_button_fontsize`, `submit_hint_fontsize`, `progress_fontsize`). N-AFC/ABX/ScaleTest font sizes scale with the window height via `_scale_factor()`.
- Area sizing: every test sizes its central area as an `[x%, y%]` percentage of the window via the same `content_area_size` key (under `ui:` for every test). All share `_BaseUIWindow._resolve_area_size()`, which clamps the percentage between a minimum and the window size minus reserved margins; the area is then centered in the window. Defaults to `[100, 100]` (max available) when the key is absent.
- Trial progress bar: every test (N-AFC, ABX, MUSHRA, ScaleTest) can show a "Trial X of Y" indicator (a text label over a slim filled bar, built by the shared `whispy.ui.base.build_progress_widget()`). Enabled per test with `show_progress` (bool) in its `ui:` block; the wording is `progress_text` (a template using `{current}`/`{total}`/`{remaining}`); the look comes from `design.yml` (`progress_bar_color`, `progress_bar_background_color`, `progress_fontsize`). The numbers travel on the `screen` dict as `progress: {current, total}` â€” injected automatically by `ExperimentScheduler` (position in the whole schedule) and `Staircase.next_screen()` (trial number, total = `max_trials` upper bound; a `progress` set by `build_screen` wins), and set explicitly by the ABX notebooks' `build_screen`. Missing/invalid progress info simply hides the bar; MUSHRA's `get_results()` excludes the `progress` key from the results columns.
- `configs/participant_id.yml`: a single `ui:` block for `whispy.ui.ParticipantID`, the window shown once before an experiment to capture a participant id (`prompt`, `submit_button_text`, `invalid_hint`; colors/fonts inherited from `design.yml`). By default it shows one free-text field (`placeholder`, optional validation `pattern`). If a `fields:` list is given (each entry an optional `{prompt, placeholder, pattern}`), it shows one labeled field per entry and the id is the answers joined by `separator` (e.g. first letters of names + a birthday digit â†’ an anonymous code). `ParticipantID(...).get_id()` returns the id. **Legacy/unused by the notebooks** (kept for possible future use): the notebooks no longer pop this window or add a `participant_id` column. Instead the participant id comes from the consent questionnaire and is baked into the result file *names* â€” see results naming below.
- `configs/welcome.yml` and `configs/thanks.yml`: a single `ui:` block each for the two info screens that frame an experiment â€” the "welcome" screen shown once before anything else (before consent) and the "thank you for participating" screen shown once after the experiment finishes. Both are rendered with `whispy.ui.InfoWindow(cfg["message"], fullscreen=cfg["fullscreen"])` and carry `message` (markdown) and `fullscreen` (bool); colors/fonts are inherited from `design.yml`. Every full-experiment notebook starts with the welcome screen and ends with the thank-you screen; `examples/building_block_welcome_and_thanks.ipynb` is the copy-paste source for both cells.
- `configs/questionnaires/`: holds all `Questionnaire` configs (`questionnaire.yml` â€” the package default/demo; `questionnaire_general.yml` â€” the experiment survey used by `examples/building_block_consent.ipynb`; `questionnaire_consent.yml`). Each splits into `ui:` (questionnaire layout/sizing; colors inherited from `design.yml`) and `questionnaire:` (list of `{section, prompt, questions: [...]}`). Supported question `type`s: `text`, `text_box`, `numeric`, `single_choice`, `multiple_choice`. `single_choice` supports an `other_question` sub-block for a free-text "other" option. Any question may carry a `depends_on` sub-block (`{question: <controlling question id>, value: <x>}` or `values: [...]`) so it is only shown when the controlling question's answer matches; hidden questions are excluded from required-answer validation and report `None` in the results.

## UI / Qt patterns to preserve

All windows in `whispy/ui/*` (`InfoWindow`, `ParticipantID`, `Questionnaire`, `DragAndDropMUSHRA`, `NAFC`, `ABX`, `ScaleTest`) follow the same shape:

- A module-level `_qapp` reference is created on first use (`QApplication(sys.argv[:1])`, only if `QApplication.instance() is None`) and kept alive for the process so windows can be constructed/destroyed repeatedly (e.g. in notebooks).
- When running inside an IPython kernel, `get_ipython().enable_gui("qt6")` is called so the Qt event loop runs alongside the kernel.
- Blocking is implemented via a private `QEventLoop` started in `wait_until_closed()`, invoked when `blocking=True`.
- Closing is gated by `_allow_close`, which is `False` unless `debug=True`. `closeEvent` ignores the close request until the gating condition is satisfied (e.g. Continue clicked, or for `DragAndDropMUSHRA`/`NAFC`/`Questionnaire`, until required interactions/answers are complete). `InfoWindow` is reused as a blocking "please complete X" prompt when validation fails.
- `InfoWindow` also maintains an `_orphaned_windows` registry so top-level info popups the caller doesn't store stay alive until closed.

## Audio / stimulus rules

- `whispy.interfaces.StimuliHandler` is an ABC with `play(stimulus)` / `stop(stimulus=None)`.
- `SoundDevice` (the default handler) loads audio via `pyfar.io.read_audio()`, rejects clipping (`max(abs(sample)) >= 1`), and requires all loaded stimuli to share one sampling rate (set as `sd.default.samplerate`).

## Results shapes

- `DragAndDropMUSHRA.get_results()` â†’ long-form `pandas.DataFrame`, one row per test stimulus, combining the screen metadata with a `rating` column (rating decoded from tile position back into the attribute's value scale).
- `NAFC.get_results()` â†’ one row per trial with `block, section, trial_id, block_name, section_name, choices, correct, selected, correct_bool, rt`.
- `ABX.get_results()` â†’ one row per trial with `block, section, trial_id, block_name, section_name, a, b, x, correct ("A"/"B"), selected ("A"/"B"), correct_bool, rt`.
- `ScaleTest.get_results(results)` â†’ long-form like MUSHRA, one row per question per stimulus with `block, section, trial_id, block_name, section_name, stimulus, question, prompt, scale_type, interaction_method, scale_min, scale_max, answer, rt` (accumulates by passing the running DataFrame back in).
- `Questionnaire.get_results()` â†’ one row per question with `section, question, prompt, type, required, answer`.

## Results saving / participant id

- `whispy/utils/results.py` (exported from `whispy.utils`) centralizes how every example notebook saves its CSV. `save_results(results, name)` writes into `examples/results/` and never overwrites (a numeric suffix is appended).
- Naming: always carries a timestamp. With a participant id â†’ `<name>_<id>_<timestamp>.csv`; without one â†’ an iterating fallback number `<name>_<NNN>_<timestamp>.csv` (`0001`..`9999`, 4 digits, raises past 9999).
- The participant id is an **in-memory value**, not a state file. The consent questionnaire (`questionnaire_consent.yml`, `pid_1..pid_4`) builds it via `participant_id_from_consent(results)`; pass that `participant_id` to `save_results(results, name, participant_id=...)` for the other blocks of the *same* notebook. The experiment building-block notebooks read it with `globals().get('participant_id')`, so pasting a consent block above wires it up automatically; standalone they fall back to the numeric form. (`whispy.utils` no longer has `set/get/clear_session_participant`; there is no `participant_id.txt`.)
- The notebooks intentionally no longer use `whispy.ui.ParticipantID` (kept as legacy) and no longer add a `participant_id` column â€” the id lives only in the file name.

## Editing conventions

- Keep new behavior config-driven, following the existing pattern of resolving default config paths via each module's `FILEPATH`.
- Use the established private-helper (`_foo`) and dataclass style seen in `whispy/ui/*` (e.g. `_MainWindow`, `_RatingArea`, `_DraggableTile`, `_DraggableTileSpec` in `drag_and_drop_mushra.py`).
- Preserve public exports in `whispy/__init__.py`, `whispy/ui/__init__.py`, and `whispy/interfaces/__init__.py` when adding new entry points.
- Prefer extending existing handlers/widgets (e.g. add a new `StimuliHandler` subclass, a new question-widget type in `questionnaire.py`, or a new attribute/config entry) over introducing ad hoc UI or audio code paths.
